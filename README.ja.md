## コンセプト

Spur（スパー）は、リプライのみのSNSのようなものです。オリジナルの投稿はなく、すべての投稿は他の投稿への返信である必要があります。これにより、最初のルート投稿から始まる木構造が作成されます。

```
         [root post]         <-- システムが作成
        /     |     \
  [reply]  [reply]  [reply]  <-- ユーザーが作成
 /       \
[reply]  [reply]   (etc.)    <-- ユーザーが作成
```

## 設計

本プロジェクトの設計は、trait（インターフェース）を介してレイヤーとモジュールを分離することに焦点を当てています。ビジネスルールは`domain`モジュールで定義され、具体的な実装は`infra`モジュールにあります。ドメインサービスは単一ドメインに固有のロジックを処理し、アプリケーションサービスは複数のドメインに跨る読み取りや書き込みのロジックを処理し、読み取りモデルは複数のドメインに跨る読み取り専用ロジックを処理します。

サーバーはOpenAPIドキュメントと統合されたSwagger UIを備えたREST APIを公開します。アプリケーション全体は、AWS EC2上でDocker Composeを使用してコンテナ化され、動作します。

```
               Caddy (HTTPSを提供)
                       |
               Swagger UI (Utoipa)
                       |
                API layer (Axum)
               /       |        \
domain services   app services   \
      \            /              \
       repositories          read models
              \                   /
               PostgreSQL database

```

## テスト

Tokioランタイム上でRust標準のテストフレームワークとMockallを使用してテストされています。一部のテストでは一時的なPostgreSQLコンテナを使用するため、実行中のDockerデーモンが必要です。

- テストの実行： `just test`
- カバレッジの表示： `just coverage`

分離された設計により、テスト戦略は`mockall`クレートによる自動モックと手動モックの組み合わせを使用した依存性注入に焦点を当てています。

テストやその他のコード品質チェックは、GitHub Actionsを使用して各プルリクエストおよびmainへのマージ時に実行され、ブランチがマージされるか、コンテナイメージがデプロイされる前に、すべてのチェックに合格する必要があります。

## セキュリティと安全性

- すべての秘密情報は環境変数を介して読み込まれます。
- アプリケーションはHTTPS経由でのみアクセスできます。
- Docker Composeスタックはルートレスで実行され、rootとしてDockerを実行することのセキュリティ上の懸念を回避します。アプリケーションコンテナ内のバイナリもルートレスで実行されます。
- スタック内のコンテナは、localhostではなく内部Dockerネットワークを介してのみ相互に公開されます。このネットワーク外へのすべての通信は、HTTPSを提供するCaddyコンテナを最初に経由する必要があります。
- ユーザーデータにアクセスするHTTPリクエストには、有効なJSON Web Tokenを含む標準のAuthorization Bearerヘッダーが必要です。
- リポジトリ全体に`unsafe`、`unwrap`、`expect`は一切含まれていません。これは`forbid`レベルに設定されたリントを使用してCIで強制されています。

## 開発とデプロイ

- プロジェクトの主要言語はRustです。Rustは[こちら](https://rust-lang.org/tools/install/)のドキュメントに従って`rustup`経由でインストールできます。
- `just`はコマンドランナーで、[こちら](https://github.com/casey/just#installation)で説明されている方法のいずれかでインストールできます。
- 開発で使用される一般的なコマンドと、それらが必要とする追加ツールは、`justfile`を介して記載されています。
- `.env.example`ファイルには、必要な環境変数の設定が記述されています。
- 開発のカンバンはGitHub Projectsの[こちら](https://github.com/users/noahkawaguchi/projects/2)にあります。
- mainへのマージ時に成功したCI/CD実行ごとに、コンテナイメージの新しいバージョンが自動的にGHCRに公開されます。そのコンテナイメージを使ってプロジェクトは[deploy/steps.md](deploy/steps.md)に記載されているようにAWS EC2にデプロイされます。
